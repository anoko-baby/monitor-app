## 1. プロジェクト概要

### 1.1 背景と目的
子ども服セレクトショップ anoko. では、モニター/モデルに商品を提供し、着用写真・動画データの提出を受けている。現在は提出期限・提出状況の管理が属人的で、提出漏れの検知が難しい。

本アプリの目的:
1. **提出漏れゼロ**: 案件ごとの期限管理と自動リマインドで未提出を防ぐ
2. **提出の一元管理**: 全提出をアプリ上で一覧・フィルタリングして確認できる
3. **ストレージ最適化**: 写真・動画の実データはDropboxに格納し、DBにはサムネイルとメタデータのみ保持する

### 1.2 ロール
| ロール | 説明 |
|---|---|
| 管理者 (admin) | Azusa。全機能+スタッフ管理・API接続設定・システム設定 |
| スタッフ (staff) | 管理者とほぼ同権限。案件作成・検収・モニター管理・一覧閲覧が可能。**不可**: スタッフアカウント管理、API接続設定、システム設定 |
| モニター (monitor) | 自分にアサインされた案件の閲覧・データ提出のみ |

### 1.3 規模想定
- 同時稼働モニター: 31〜100人
- 月間提出数: 想定100〜400件
- 対応言語: 日本語のみ / タイムゾーン: JST固定

## 2. 技術スタックと全体アーキテクチャ

### 2.1 スタック(確定)
| レイヤ | 技術 | 備考 |
|---|---|---|
| モバイルアプリ | **Expo (React Native) + TypeScript** | Expo Router使用。iOS/Android両対応 |
| バックエンド | **Supabase** | Auth / Postgres / Storage(サムネイルのみ) / Edge Functions |
| 定期処理 | Supabase Cron (pg_cron) + Edge Functions | リマインド判定・送信 |
| ファイル格納 | **Dropbox API** | 写真・動画の実データ。チャンクアップロード対応 |
| 商品マスタ・注文連携 | **Shopify Admin API** (カスタムアプリ / read_products・read_orders・read_customers・read_discounts) | 案件登録時の商品検索・注文取込、クーポン注文の検知 |
| 認証(モニター) | **招待コード + メールアドレス/パスワード** (Supabase Auth) | 招待コードで仮登録レコードと紐付け、本登録はメール/パスワード |
| 認証(管理者/スタッフ) | メールアドレス+パスワード (Supabase Auth) | |
| 通知 | **Expo Push Notifications のみ** | LINE連携なし。モニターは通知許可を必須ステップとして求める |
| ビルド/配信 | EAS Build / EAS Submit | |

### 2.2 アーキテクチャ原則
- モバイルアプリからDropbox/Shopifyのシークレットに直接触れない。**外部API呼び出しはすべてSupabase Edge Functions経由**とし、トークンはSupabase側で管理する。
- DBアクセスはSupabase RLS(Row Level Security)で保護。モニターは自分のデータのみ参照可能。
- 写真・動画の実体はDropboxのみ。DBにはDropboxパス・共有リンク・サムネイルURL・メタデータを保存。

### 2.3 配信戦略(重要)
自社モニター限定アプリのため、App Store一般公開審査(ガイドライン3.2)で却下されるリスクがある。
- **iOS**: Unlisted App Distribution(非公開リンク配信)を申請。承認まではTestFlightで運用開始可
- **Android**: Google Play「限定公開(非公開テストまたはURL限定)」で配信
- 必要アカウント: Apple Developer Program(年 約15,800円)、Google Play Console(初回 約3,700円)

## 3. 機能要件

### 3.1 認証・アカウント

#### 3.1.1 モニター(招待制+メールアドレス/パスワード)
1. 管理者/スタッフがアプリでモニターを仮登録(氏名・ニックネーム)→ **8桁英数の招待コード**を発行(有効期限14日、ワンタイム)
2. モニターにアプリのインストール案内と招待コードを送付(運用はLINE公式アカウント等、アプリ外。このアプリ自体はLINEと連携しない)
3. モニターがアプリを起動 → 招待コード入力 → メールアドレス・パスワードを設定して本登録(Supabase Auth) → 仮登録レコードと紐付き本登録完了
- 招待コード無効/期限切れ時はエラー表示のみ(自由登録は不可)
- **パスワードを忘れた場合**: Supabase Auth標準のパスワードリセット(メール送信)を使用
- **機種変更時**: 同じメールアドレス・パスワードで新しい端末からログインすれば、案件・提出履歴はそのまま維持される(招待コードの再発行は不要)。メールアドレス自体を変更したい場合のみ管理者/スタッフに依頼しプロフィール編集で変更する

#### 3.1.2 管理者・スタッフ
- メールアドレス+パスワード(Supabase Auth)。初期管理者はシード登録
- 管理者のみ、設定画面からスタッフアカウントの追加・無効化が可能

#### 3.1.3 モニタープロフィール
- 氏名 / ニックネーム / メールアドレス / 通知許可状態(表示のみ。アプリ内に通知をOFFにする設定項目は設けない。詳細は3.8)
- 子ども情報(複数登録可): 呼び名 / 生年月 / 性別(任意)
- 掲載許諾フラグ: EC掲載 / SNS掲載 / 広告利用(各 可・不可。本登録時に同意画面で取得し、管理者が後から編集可能)
- 住所・電話番号は**アプリでは保持しない**(発送はShopify/既存フローで管理。個人情報最小化)

### 3.2 商品マスタ(Shopify連携) 【Phase1】
- Shopifyカスタムアプリを作成。スコープは `read_products` / `read_orders` / `read_customers` / `read_discounts`(注文・顧客・クーポンは3.12のクーポン注文連携で使用。書き込み権限は一切付与しない)
- 案件登録画面に商品検索(キーワード)を設置。Edge Function経由でShopify Admin GraphQL APIを呼び、商品・バリアント(SKU/サイズ/カラー/画像)を取得
- 選択した商品・バリアントはローカルの `products` / `variants` テーブルにキャッシュ(スナップショット)して案件に紐付ける(Shopify側で商品が削除されても案件表示が壊れないようにするため)
- **手動入力フォールバック**: Shopify未登録品(サンプル等)は商品名・ブランド・サイズ・カラーを手入力で案件登録できる

### 3.3 案件(キャンペーン)管理 ※管理者/スタッフ

#### 3.3.1 案件登録項目
| 項目 | 必須 | 内容 |
|---|---|---|
| 案件番号 | 自動 | 登録時に自動採番(`A-0001` 形式)。変更不可。検索・Dropboxフォルダ名に使用 |
| 案件名 | ○ | 例:「Organic Zoo 26SS モニター(山田様)」自動生成候補あり |
| モニター | ○ | 登録済みモニターから選択 |
| 対象の子ども | - | モニターの子どもから選択(任意) |
| **Shopify注文の取込** | - | モニター品発送時に起票した注文を注文番号で検索 → **商品SKU・発送日を自動入力**し、注文の顧客IDをモニターのプロフィールへ自動保存(紐付け済みの場合は上書きしない)。取込を使わない手動登録も引き続き可 |
| 商品(複数可) | ○ | Shopify検索 or 手動入力。SKU単位(サイズ/カラー確定) |
| 商品発送日 | - | 日付 |
| 到着確認 | 自動 | モニターが「受け取りました」をタップした日時を記録 |
| 提出期限設定 | ○ | 単発 or 繰り返し(3.3.2) |
| SNS投稿必須 | ○ | ON/OFF。ONの場合は頻度と投稿期限を設定(3.3.3) |
| 提出フォーム項目 | ○ | 項目マスタからON/OFF選択(3.4.2) |
| 撮影ガイドライン | - | テキスト(改行可・1000字まで)。モニターの案件詳細・提出画面に表示 |
| リマインド | ○ | ON/OFF(デフォルトON) |
| 社内メモ | - | モニターには非表示 |

- 案件はアサイン(登録)と同時に開始される。モニター側の「引き受ける」確認ステップは設けない(モニターとの合意はLINE等アプリ外で取得済みの前提)

#### 3.3.2 期限設定(データ提出タスク)
- **単発**: 提出期限日を1つ指定
- **繰り返し**: 「開始月 / 毎月◎日まで / 全n回」を指定 → 登録時に **n個の回次(cycle)を一括生成**する(例: 2026年8月開始・毎月10日・全6回 → 8/10, 9/10, ..., 1/10 の6回次)
- 月末日指定(29〜31日)は存在しない月では**その月の末日**に丸める
- 期限は当日 23:59 (JST) まで有効
- 生成後、各回次の期限は個別に編集可能(差し戻しや事情による延長に対応)

#### 3.3.3 SNS投稿タスク
- 「SNS投稿必須」ONの場合、頻度を選択:
  - **毎回**: 各回次にSNS投稿タスクを生成。投稿期限は「毎月◎日」を別途指定(データ期限と独立)
  - **案件全体で1回**: 案件に1つだけSNS投稿タスクを生成。期限日を1つ指定
- SNS投稿タスクはデータ提出タスクと**独立したステータス・期限・リマインド**を持つ
- 同一回次で「データ提出済み・投稿未提出」等の片方だけの状態が成立し、一覧で判別できること

#### 3.3.4 案件の操作
- **複製**: 既存案件をコピーして新規作成。商品・期限設定・SNS設定・フォーム項目・撮影ガイドラインを引き継ぎ、モニター(と発送日)だけ差し替えて登録できる(同一商品を複数モニターへ配る運用向け)
- **一括延長**: 未提出の残り回次すべての期限を「+n日ずらす」または「毎月◎日を変更」でまとめて変更。変更時はモニターへ通知
- **途中中止**: 指定回次以降の未提出タスクを `cancelled` にする。キャンセル済みタスクはリマインド対象外・一覧のデフォルト表示から除外。案件全体の中止は案件ステータス `cancelled` で行う

### 3.4 データ提出(モニター)

#### 3.4.1 提出フロー
1. ホームの「あなたの案件」から案件を選択(複数案件対応)
2. 回次一覧から提出する回を選択 → 提出フォームへ
3. 写真・動画を選択してアップロード + フォーム項目入力 → 提出
- 提出は**期限まで追加・差し替え可能**(提出済み後も期限内は編集可。編集時は再提出扱いにせずファイル追加のみ許可し、検収済み後は編集不可)
- アップロード上限(app_settingsで変更可能な設定値として実装):
  - 写真: 1回次あたり最大30枚、1枚50MBまで(HEIC/JPEG/PNG)
  - 動画: 1回次あたり最大5本、1本2GBまで(MOV/MP4)
- アップロードは非同期実行・進捗表示・失敗時リトライ(3回)・中断再開(Dropboxアップロードセッション利用)。**Phase1ではアプリを開いたまま行う前提**とし、OSレベルのバックグラウンド実行(アプリ切替後・画面消灯後も継続)は対象外とする。通信断(機内モード等)からの再開には対応する
- **オリジナル画質のままアップロード**: 端末側での再圧縮・リサイズは行わない(HEICは元解像度維持。モデルデータのため画質劣化は不可)
- **位置情報の除去**: 写真・動画のGPS情報(EXIF位置データ)はアップロード時に自動除去する(撮影日時などその他のメタデータは保持)
- モニター設定「Wi-Fi接続時のみアップロード」ONの場合、モバイル回線では送信キューに保持しWi-Fi接続時に自動送信

#### 3.4.2 提出フォーム項目マスタ
案件登録時に必要な項目のみONにする方式。撮影日のみ固定必須。

| key | 項目 | 型 | 備考 |
|---|---|---|---|
| shot_date | 撮影日 | date | **固定・常に必須** |
| height | 身長 | number(cm) | |
| weight | 体重 | number(kg) | |
| age_months | 月齢/年齢 | number(ヶ月)+年齢併記表示 | 子ども情報の生年月から自動計算し初期値表示 |
| foot_size | 足の実寸 | number(cm) | シューズ案件用 |
| fit | フィット感 | 選択式 | 小さい / ぴったり / やや大きい / 大きい |
| memo | ひとことメモ | text(500字) | |

- 項目マスタはDBテーブルで管理し、**将来管理者が項目を追加できる**拡張性を持たせる(型: date / number / select / text)
- 着用サイズは入力させない(案件に紐付いたSKUのサイズ/カラーを提出詳細に自動表示する)

#### 3.4.3 SNS投稿記録の提出
- 投稿URL(必須・URL形式バリデーション)+ 任意メモ
- 複数投稿があれば複数URL追加可(最大5件/タスク)

### 3.5 検収・差し戻し(管理者/スタッフ)
- 提出詳細画面で内容・サムネイル・Dropboxリンクを確認し、以下いずれかを実行:
  - **確認済みにする** → タスク完了
  - **差し戻す** → 差し戻しコメント(必須)+ 新しい提出期限(必須)を設定 → モニターへ通知
- 差し戻されたタスクはモニター側で「差し戻し」表示となり、同じフォームから再提出できる。再提出でステータスは「提出済」に戻る
- 差し戻し履歴(コメント・日時・担当者)は提出詳細に時系列表示

### 3.6 一覧・フィルタリング

#### 3.6.1 モニター側
- ホーム: 自分の案件一覧(期限が近い順)。各案件に「次の期限」「未提出バッジ」を表示
- 提出履歴: 自分の全提出を時系列一覧(サムネイル付き)

#### 3.6.2 管理者/スタッフ側
- **全提出一覧**: 全モニターの全タスクを一覧表示
- フィルタ: 期間(期限日/提出日) / モニター / **モニタータグ**(Phase2) / ブランド / 商品 / ステータス(未提出・提出済・確認済・差し戻し・期限超過) / タスク種別(データ/SNS)
- キーワード検索: **案件番号**(A-0123)・案件名・モニター名・商品名で横断検索。案件番号はDropboxフォルダ名と一致するため相互に探しやすい
- **採用マーク★**: 提出詳細のサムネイルをタップして★ON/OFF(管理者/スタッフのみ)。一覧・詳細に「★のみ表示」フィルタを設ける(SNS素材の候補選定用)。将来のCSV出力追加を見据え、提出一覧の取得クエリは1つのビュー/エンドポイントに集約して実装すること
- ソート: 期限日 / 提出日 / モニター名
- 各行の表示: サムネイル(先頭1枚)、モニター名、商品名+SKU、回次(第n回/全m回)、データ📷とSNS📱それぞれのステータスアイコン、期限
- 行タップ → 提出詳細(全サムネイルグリッド、フォーム回答、**Dropboxフォルダを開くリンク**、検収操作)

### 3.7 ダッシュボード(管理者/スタッフのホーム) 【Phase2】
- 期限超過(未提出のまま期限を過ぎたタスク)件数と一覧ショートカット
- 今後7日以内に期限を迎える未提出タスク一覧
- 検収待ち(提出済)件数
- 進行中案件数 / 稼働モニター数
- Phase1では管理者/スタッフのホームは案件一覧・全提出一覧への簡易ナビゲーションのみとする

### 3.8 通知仕様 【Phase1・Push通知のみ】

| # | トリガー | 宛先 | チャネル |
|---|---|---|---|
| N1 | 案件アサイン(登録)時 | モニター | Push |
| N2 | リマインド: 期限**7日前**と**前日**の朝9:00(JST)、対象タスクが未提出の場合のみ | モニター | Push |
| N3 | 差し戻し時 | モニター | Push |
| N4 | 確認済み(検収完了)時 | モニター | Push |
| N5 | モニターが提出した時 | 管理者/スタッフ | Push |
| N6 | 期限超過の督促: **期限の2日後** 朝9:00、なお未提出の場合 | モニター | Push |
| N7 | 期限超過の発生報告(N6と同時) | 管理者/スタッフ | Push |
| N8 | ~~LINE送信控え~~ | ― | **v1.8で廃止(LINE連携なし)** |
| N9 | お知らせ配信時(3.9) | 対象モニター | Push |
| N10 | 到着確認リマインド: 発送日から**5日**経過しても到着確認が押されない場合(日数は設定値)。管理者/スタッフにも同時に報告 | モニター(報告は管理者/スタッフへPush) | Push |
| N11 | 監視対象クーポンの新規注文を検知した時(3.12) | 管理者/スタッフ | Push |
| N12 | 外部連携の異常検知: Dropboxトークン失効 / Shopify APIの連続エラー / Dropbox容量80%超過 | 管理者 | Push |

- リマインドは案件単位のON/OFF設定に従う。データタスクとSNSタスクは独立して送信
- リマインド日程(7日前/前日)と超過督促(+2日)は `app_settings` の設定値として実装し、後から変更可能
- 通知チャネルはExpo Push Notificationsのみ。**モニターは初回登録時のオンボーディングで通知許可を必須ステップとして求める**(許可するまで次へ進めない導線。ただしOS側で「許可しない」を選んだ場合はブロックできない)。アプリ内には通知を個別にOFFにする設定項目を設けない
- 通知許可がOS側で無効化されている場合、モニターのホーム画面に有効化を促す常時バナーを表示し、タップでOS設定画面へ遷移させる
- 通知文面はテンプレート化し `notification_templates` で管理(変数: モニター名/案件名/商品名/期限日/回次)
- 送信結果は `notification_logs` に記録(監査・再送判定用)

### 3.9 お知らせ配信(管理者/スタッフ) 【Phase1: 全モニター/個別選択のみ。タグ・グループ指定はPhase2】
- 配信先は選択: **全モニター / 個別選択**(タグ・グループ指定(3.11)はPhase2で追加)
- 内容: タイトル+本文+**リンクボタン1つ(ラベル+URL、任意)**。モニター限定クーポン配布時は、本文にクーポンコードを記載しリンクボタンでショップやLINEクーポンページへ誘導する運用を想定
- 配信と同時にPush通知(N9)。送信前に**対象人数と対象者一覧のプレビュー**を必ず表示し、確認後に送信
- モニター側は「お知らせ」画面で一覧・詳細を閲覧。未読バッジを表示
- テーブル: `announcements`(title / body / link_label / link_url / target_type[all/manual/segment] / segment_id / created_by / sent_at)、`announcement_targets`(announcement_id / monitor_id / read_at)※送信時点で対象を確定して記録。`target_type=segment` はPhase2でモニタータグ・グループ(3.11)実装後に有効化する

### 3.10 モニター実績(モニター詳細画面) 【Phase1】
- **提出率**: 完了(approved+submitted)タスク ÷ 対象タスク(cancelled除く)
- **期限内提出率**: `first_submitted_at ≤ due_date` だったタスクの割合
- 過去案件履歴(案件番号・商品・期間・完了状況)を時系列表示
- 実績はモニター本人には表示しない(管理者/スタッフのみ)

### 3.11 モニタータグとグループ(管理者/スタッフ) 【Phase2】
#### タグ
- 管理者/スタッフがタグマスタを自由に作成(名称+色)。モニターに複数付与できる
- 用途例: 「ベビー」「キッズ」「シューズOK」「動画が得意」「広島近郊」「殿堂入りモニター」など属性・実績の自由なラベリング
- モニター一覧をタグでフィルタ可能。モニター詳細画面でタグの付け外しができる
- タグはモニター本人には表示しない(内部管理用)

#### グループ(保存セグメント)
- **タグの組み合わせ+条件**で定義し、名前を付けて保存する
  - 条件: 「選択タグを**すべて**含む(AND)」/「選択タグの**いずれか**を含む(OR)」
  - 例: グループ「ベビー動画班」= タグ[ベビー]AND[動画が得意]
- グループの所属は**動的評価**: お知らせ送信時点のタグで対象者を決定する(タグを付け替えるだけでグループの中身が自動更新され、メンテナンス不要)
- お知らせ配信(3.9)の配信先として選択可能。将来的に案件一覧・提出一覧のフィルタにも利用できる構造とする
- テーブル: `tags`(name / color / sort_order)、`monitor_tags`(monitor_id / tag_id)、`segments`(name / tag_ids jsonb / match_type[all/any])

### 3.12 クーポン注文の検知・案件化(Shopify注文連携) 【Phase1】
目的: 過去モニターへ「モニター価格クーポン」をグループ配信 → クーポン使用注文を自動検知 → データ提出義務のある**案件としてワンタップ登録**、までをアプリ内で完結させる。Phase1時点ではグループ配信(3.11)が未実装のため、クーポン配布自体はアプリ外(LINE公式アカウント等)で行い、**注文の検知・案件化のみ**をアプリで扱う。

#### 監視対象クーポンの登録
- Shopifyで作成済みのディスカウントコードのうち「モニター限定クーポン」をアプリに登録する。登録済みコードを使った注文だけが検知対象
- コード登録時にShopify APIで実在チェックを行う

#### 注文の検知とモニター紐付け
- Shopify Webhook `orders/create` をEdge Functionで受信(HMAC署名検証必須)し、監視対象コード使用の注文のみ保存。取りこぼし防止に1日1回の照合ポーリングも併用
- 保存内容: 注文番号 / 注文日時 / クーポンコード / 顧客名・メール(スナップショット)/ Shopify顧客ID / 商品明細(SKU・サイズ・カラー・数量)
- **紐付けロジック**: 注文のShopify顧客IDが `profiles.shopify_customer_id` と一致すれば自動紐付け。未一致は「未紐付け」と表示し、管理者がモニターを手動選択 → その際に顧客IDをプロフィールへ保存し、**次回以降は自動紐付け**される。なお案件登録時にShopify注文取込(3.3.1)を運用していれば、モニターは初回案件の時点で紐付け済みになるため、手動紐付けはほぼ発生しない想定

#### 「クーポン注文」タブ(管理者/スタッフ)
- 一覧表示: 注文番号 / 日付 / 顧客名 / モニター(紐付け状態)/ クーポンコード / 商品 / 対応状況
- 対応状況: **未対応 / 案件化済み / 対象外**(データ提出を求めない注文は「対象外」でスキップ既読化)
- **「案件化する」** → 案件作成画面(3.3)へ遷移し、モニター・商品SKU・案件名を注文内容から自動入力(期限・SNS設定・フォーム項目を設定して保存)。作成後は注文⇄案件を相互リンク(campaigns.source_order_id)
- 新規のクーポン注文を検知したら管理者/スタッフへ通知(N11)

#### 個人情報の扱い
- 保存するShopify顧客情報は氏名・メール・顧客IDの最小限に留める(住所・電話番号は取得しない)
- テーブル: `watched_coupons`(code unique / label / active)、`coupon_orders`(shopify_order_id unique / order_no / ordered_at / coupon_code / customer_name / customer_email / shopify_customer_id / line_items jsonb / monitor_id nullable / status[pending/converted/skipped] / campaign_id nullable)

## 4. ステータス定義と遷移

### 4.1 タスクステータス(データ提出タスク / SNS投稿タスク共通)
```
pending(未提出)
   │ モニターが提出
   ▼
submitted(提出済)──管理者が確認済み──▶ approved(確認済)【終端】
   │
   └─管理者が差し戻し(コメント+新期限)──▶ rejected(差し戻し)
                                              │ モニターが再提出
                                              ▼
                                          submitted(提出済)に戻る
```
- 派生表示ステータス「**期限超過(overdue)**」: `status = pending または rejected` かつ `期限 < 現在時刻` の場合に一覧・フィルタで表示(DBには持たず算出)
- `cancelled`: 途中中止でスキップされた回次のタスク。リマインド・実績集計の対象外
- **期限超過後も提出は可能**(ブロックしない)。`first_submitted_at > due_date` の提出は一覧・詳細で「期限超過提出」バッジを表示

### 4.2 案件ステータス
- `active`(進行中)→ `completed`(全タスクapprovedで自動遷移、手動も可)/ `cancelled`(中止・手動)

## 6. Dropbox連携仕様

### 6.1 フォルダ命名規則(商品別→モニター別)
```
/anoko_monitor/{ブランド名}/{商品名}_{SKU}/{案件番号}_{モニター名}/第{n}回_{提出期限YYYYMMDD}/
例: /anoko_monitor/OrganicZoo/StripeSweater_OZ-SS-6-12M/A-0123_山田花子/第2回_20260910/
```
- パス使用不可文字(/ \ : * ? " < > |)は「-」に置換。同名衝突時は末尾に `_2` 等を付与
- 複数商品の案件は先頭商品のフォルダを使用し、案件の `dropbox_base_path` に確定パスを保存
- ファイル名: `{回次}_{連番}_{元ファイル名}`(例: `02_001_IMG_1234.jpg`)
- 期限を一括延長・個別変更してもフォルダ名の日付は**リネームしない**(作成時の期限で固定。保存済みリンクの断絶を防ぐ)

### 6.2 アップロード方式
- Edge Function がDropboxの**短命アクセストークンを発行**し、アプリは実データをDropboxへ直接アップロードする(Supabaseを経由させず転送コストと制限を回避)。Dropboxのリフレッシュトークン・App Secretはクライアントに一切渡さない
- 150MB超のファイルは Dropbox `upload_session`(チャンクアップロード、8MB単位)を使用。中断時はセッションIDから再開
- アップロード完了後、Edge Function が共有リンク(shared link)を生成して `submission_files` に保存
- Dropbox認証: Scoped App(files.content.write / files.content.read / sharing.write)+ リフレッシュトークンをSecretsに保存

### 6.3 サムネイル
- アプリ側(クライアント)で生成: 写真=長辺400pxのJPEG / 動画=先頭1秒のフレーム画像
- Supabase Storage の `thumbnails` バケットに保存(1枚あたり数十KB想定 → 月400提出×平均10ファイルでも軽微)

## 9. 非機能要件

- **対応OS**: iOS 16以上 / Android 10以上
- **性能**: 一覧表示は50件ずつページネーション。画像はサムネイルのみ読み込み
- **セキュリティ**: 全テーブルRLS必須 / APIシークレットはEdge Function Secretsのみ / 通信はHTTPS / approved後の提出データは編集不可
- **オフライン**: 提出フォームの入力値は端末にドラフト保存(アップロード失敗時に消えない)
- **バックアップ**: Supabase自動バックアップ(日次)。Dropboxは実データそのものが保管場所
- **監査**: 検収操作・通知送信はログテーブルに記録
- **強制アップデート**: 起動時に `app_settings.min_app_version` をチェックし、下回る場合は更新案内画面でブロック
- **法務**: 利用規約・プライバシーポリシーはストア審査要件。アプリ内表示+Web公開URLの両方を用意
- **削除方針**: 案件・提出・モニターの削除は論理削除(deleted_at)とし、一覧から非表示にするのみ。Dropbox上の実ファイルはアプリからは削除しない(完全削除が必要な場合はDropbox上で手動)
- **保存ポリシー**: 提出データは無期限保存(自動アーカイブなし)。Dropbox容量の使用率をダッシュボードに表示し、80%超過で管理者に通知(N12)
- **連携ヘルスチェック**: 外部APIの認証失効・連続エラーを日次で検査し、異常時は管理者に通知(N12)。アップロードが静かに失敗し続ける事故を防ぐ

## 13. 用語定義
| 用語 | 定義 |
|---|---|
| 案件(campaign) | 1モニター×1商品セットのモニター契約単位 |
| 回次(cycle) | 繰り返し案件の各回(第n回)。単発案件は回次1つ |
| タスク(task) | 回次×種別(データ提出/SNS投稿)の提出単位。ステータス・期限・リマインドを持つ |
| 検収 | 管理者/スタッフによる提出内容の確認(確認済 or 差し戻し) |

## 14. UIデザイン指針

### 14.1 デザインコンセプト
「**生成りの服に付いた、上質な織りネームタグ**」。anoko.が扱うオーガニック子ども服の世界観——リネンや生成りの布、くすんだアースカラー、余白のあるInstagramの世界——をそのままアプリに持ち込む。装飾は最小限、余白と文字組みで上質さを出す。派手なグラデーション・強い影・ポップな原色は使わない。

### 14.2 カラートークン(実装時はこの値をそのまま使用)
| トークン | 値 | 用途 |
|---|---|---|
| bg | #F6F3ED | 画面背景(生成り) |
| surface | #FFFFFF | カード・入力欄 |
| ink | #3E3A34 | 基本文字(墨茶。真っ黒は使わない) |
| ink-soft | #8C8579 | 補助文字・ラベル |
| line | #E7E1D6 | 罫線・カード枠(影の代わりにヘアライン枠を基本とする) |
| accent | #7E8F86 | 主アクション・選択状態(くすみセージ) |
| accent-ink | #4E5B54 | accent上の濃色/押下状態 |
| status-submitted | #8FA3B5 | 提出済(ダスティブルー) |
| status-approved | #7E8F86 | 確認済(セージ) |
| status-rejected | #C9A227 抑えめ→実装値 #B79A4B | 差し戻し(からし) |
| status-overdue | #B9705F | 期限超過(れんが) |
| status-pending | #B7AE9E | 未提出(グレージュ) |

- ステータス色は全て低彩度で統一。バッジは「淡色背景(色の12%不透明)+同系濃色文字」のピル型
- ダークモードはv1では非対応(OS設定に関わらずライト固定)

### 14.3 タイポグラフィ
| ロール | フォント | 用途 |
|---|---|---|
| 見出し | **Zen Maru Gothic**(Medium) | 画面タイトル・案件名。丸みで子ども服らしい柔らかさを出す |
| 本文 | **Noto Sans JP**(Regular/Medium) | 本文・フォーム |
| 数字・日付・案件番号 | Noto Sans JP の等幅数字(tabular-nums) | 期限日・A-0123・回次。桁が揃い一覧が読みやすい |

- 型スケール: 22 / 17 / 15 / 13 / 11pt。行間はゆったり(本文1.6)。文字を詰めない
- 期限日は一覧の中で最も目立つ要素とする(サイズと色で優先度を表現し、アイコンを増やさない)

### 14.4 シグネチャー要素: スナップボタン・ドット
繰り返し案件の回次進捗を、**ベビー服のスナップボタン(ドット)**をモチーフにした横並びドットで表現する。
- 完了(確認済)= accentで塗りつぶし+中央に小さな凹み(スナップの窪み)/ 提出済 = 塗り+リング / 未提出 = line色の輪郭のみ / 期限超過 = status-overdueの輪郭 / キャンセル = 打ち消し斜線
- 案件カード・案件詳細・提出一覧の行に共通で使用し、アプリ全体の記憶に残る要素はこれ一つに絞る。他の装飾は加えない

### 14.5 コンポーネント規則
- 角丸: カード16 / ボタン・入力12 / バッジ全丸。影は使わず `line` の1pxヘアライン枠+背景色差で面を分ける
- 主ボタンはaccent塗り+白文字、副ボタンは枠線のみ。1画面に主ボタンは1つ
- アイコン: lucide系ラインアイコン(stroke 1.5)。多用せず、ステータスは色+文字を優先
- タップ領域は最小44pt。フォームは1画面1テーマ(入力項目を詰め込みすぎない)
- サムネイルグリッドは正方形・間隔2px・角丸8。写真が主役になる画面ではUI色を極力引く
- 空状態はイラストを置かず、短い一文+次の行動ボタンのみ(例:「まだ提出はありません」+「提出する」)
- モーション: 画面遷移はOS標準、状態変化は150msのフェードのみ。祝い演出は提出完了時のトースト1回に留める

### 14.6 文言トーン
- モニター向けは「です・ます」の柔らかい短文。命令形を避ける(「提出してください」→「◎月◎日までにお願いします」)
- ボタンは動作をそのまま書く:「提出する」「受け取りました」「差し戻す」。システム用語(タスク/ステータス等)はモニター画面に出さず「提出」「確認」等の日常語に置き換える
- エラーは原因+対処を1文ずつ。謝罪文は入れない

### 14.7 実装方法
- NativeWind(Tailwind for RN)を採用し、14.2〜14.5のトークンを `tailwind.config.js` の theme として一元定義。画面側での直値指定(hexベタ書き)は禁止
- フォントは expo-font でZen Maru Gothic / Noto Sans JPを同梱
